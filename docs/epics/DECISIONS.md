# Decisions

Lightweight log of cross-cutting decisions referenced by the epics/stories.
Status: **accepted** (decided) · **proposed** (leaning, needs validation) ·
**open** (undecided, needs a spike). Link decisions from stories as `D-NN`.

| ID | Decision | Status |
| --- | --- | --- |
| D-01 | **Target app is MIDI Maze.** The supported class is ST software that does MIDI I/O through the OS (`Midiws`, `Bconout`/`Bcostat`, `Bconin`/`Bconstat`, possibly `Iorec`). Apps that talk to the 6850 MIDI ACIA registers (`$FFFC04`/`$FFFC06`) directly and install their own MIDI interrupt handler are **out of scope** — those bus cycles never reach the cartridge, so this approach cannot intercept them. _Confirmed by the project's NotebookLM source: MIDI Maze accesses MIDI via BIOS/XBIOS (trap #13/#14) Device 3 as a simple serial port._ | accepted |
| D-02 | **The MIDI stream is an opaque raw byte pipe.** MIDI Maze uses MIDI as a dumb byte transport, so we forward bytes byte-by-byte / buffered with **no MIDI parsing**: no message framing, no running-status handling, no active-sensing/clock filtering, no RTP-MIDI. Byte-exact, order-preserving passthrough. Any "be smart about MIDI" optimization would corrupt the game. _Confirmed by source: MIDI Maze sends raw 8-bit bytes (e.g. `0x00` designates the ring MASTER), not valid MIDI messages._ | accepted |
| D-03 | **Network transport = TCP with `TCP_NODELAY`.** The raw byte ring is order- and integrity-sensitive — a dropped/reordered byte desyncs the game; UDP would mean reinventing sequencing/retransmit. Stack fit: the Pico-W firmware already runs lwIP TCP (httpc/download), and the Python orchestrator serves many concurrent player sockets cleanly with `asyncio` TCP. `TCP_NODELAY` defeats Nagle to protect latency (C-01); revisit only if EPIC-07 shows it insufficient. | accepted |
| D-04 | **Topology = star via a central orchestrator that emulates the ring.** Each Atari + Pico-W connects to a Python orchestrator (**in this repo** under `orchestrator/`, EPIC-04 — revises the earlier "separate project" plan). The orchestrator is a **ring-order router, not a broadcaster**: it wires OUT(player N) → IN(player N+1) preserving order so the protocol works naturally (master election via `0x00` returning, `0x80` COUNT-PLAYERS incrementing around the loop). **Ring membership must be fixed before a game starts** — you can't add/drop a player mid-game without breaking COUNT-PLAYERS and the lock-step (C-01). This implies EPIC-03's transport needs ordered, per-peer delivery, not broadcast. MVP = 2 players through the orchestrator; the full ring (up to 16 entities, humans + drones) is post-MVP. The firmware-side interface is specified in [`ORCHESTRATOR-CONTRACT.md`](ORCHESTRATOR-CONTRACT.md); the orchestrator lives in this repo under `orchestrator/` (EPIC-04, Python 3 stdlib-only). | proposed |
| D-05 | **MIDI path = BIOS Device 3 + an XBIOS MIDI-IN read after every write.** Per thesis §2.4.2 (verified in the Steem debugger) and Annex B figs B.1–B.40: MIDI Maze **sends** via BIOS `Bconout(3)` (`trap #13` fn 3, `jsr $341b2`); **polls/reads** input in the game loop via BIOS `Bconstat(3)`/`Bconin(3)` (fns 1/2, `and.l #$ff,d0`); and **after every write** reads MIDI IN back via an **XBIOS (`trap #14`) call** through wrappers `$188f0`→`$341a2`, byte returned in D0 — this is the ring readback that drives master election (`tst d0`==0 → MASTER) and the per-frame state exchange. So **both trap #13 and trap #14 are on the critical path**; the XBIOS read is essential, not auxiliary — if it isn't serviced, MIDI Maze's send routines take their error path and abort. The exact XBIOS fn# isn't named in the source (the author describes the behaviour; plausibly `Iorec(2)` = XBIOS 14, since BIOS `Bconin` and XBIOS `Iorec` read the same system MIDI input buffer). Practical approach: feed received bytes into that system input buffer so both read paths see them, and hook `Bconout(3)` for output. `Midiws` (XBIOS 12) is not used. Source: <https://zaguan.unizar.es/record/146840/files/TAZ-TFG-2022-4616.pdf> §2.4.2 + Annex B. **Confirmed on hardware:** a tracer caught the first `Bconout(3)` byte = `0x00` (the master-election request), and a loopback feeding the `Iorec(2)` MIDI input buffer let MIDI Maze read its own output back and become MASTER — so input is read from the system MIDI input record (Bconin/readback), exactly as feeding `Iorec` predicts. | accepted |
| D-06 | **Validation harness is cartridge-resident.** The self-test runs as m68k code in the cartridge at boot (gated to a test build), not a standalone `.PRG`. | accepted |
| D-07 | **Physical-MIDI passthrough (THRU to the ST's own MIDI port) is deferred.** MIDI-to-IP replaces the physical link for MIDI Maze; local echo to the ACIA is not in the alpha scope. | accepted |
| D-08 | **Orchestrator handles raw bytes over sockets — never via a host MIDI driver.** A documented prior attempt to bridge the ST to a PC failed because host MIDI stacks (Windows `WINMM.dll`) filter/block non-standard raw single-byte traffic, with no API to pass a lone raw byte. Our design sidesteps this: the Pico-W intercepts at the Atari BIOS/XBIOS layer and ships raw bytes over TCP to a Python (`asyncio`) socket server. The orchestrator treats the stream as opaque bytes; it must not route through any OS MIDI API. | accepted |
| D-09 | **MIDI Maze needs ≥2 players; a loopback can't start a game.** Confirmed on hardware: the MASTER reaches the config screen and then waits for a SLAVE to join the ring — a single node never starts a match. So the EPIC-01/02 loopback validates the *transport* (master election + COUNT-PLAYERS + config round-trip), not *gameplay*. Real gameplay needs a 2nd node: the orchestrator (EPIC-04) wires the ring, and the 2nd node is another ST+RP or — with no hardware — a **Hatari gateway player** (EPIC-05). The RP/byte-pipe stays protocol-dumb (D-02); any future "fake SLAVE"/drone is MIDI-Maze-aware logic that lives in the orchestrator (a later smart epic), never the RP. | accepted |
| D-10 | **Server + Hatari gateway live in this repo, Python 3 stdlib only.** The orchestrator (`orchestrator/`, EPIC-04) and the Hatari gateway (`hatari-gateway/`, EPIC-05) are written in Python 3 using only the standard library — no third-party packages, ever (`asyncio`, `socket`, `http.server`, `os.mkfifo`). The orchestrator MVP is a **dumb ring relay with low-level connection tracking** — no MIDI-Maze protocol parsing; game-session detection ("gameplays disputing/disputed") is a later smart epic. The Hatari gateway is a **software RP2040** (file-FIFO ↔ orchestrator) and the project's test peer (replaces the old desktop-test-peer story). | accepted |
| D-11 | **A real match needs a protocol-aware coordination layer in the orchestrator (EPIC-08); the dumb relay (D-10) is necessary but not sufficient.** Hardware testing (EPIC-05), once transport and latency were fixed, surfaced two blockers a byte-dumb relay can't solve: (a) **master election doesn't converge** when machines launch asynchronously — multiple nodes auto-elect, a SLAVE re-sends every `0x00` (Anexo A), and with no single master the election byte loops forever or re-triggers mid-`COUNT-PLAYERS`, so the MASTER reads a garbage/zero player count and never starts; (b) **the ~4 KB `SEND-DATA` maze (`0x83`) overflows the RP's IN path** (1 KB queue / 256 B `MIDI_IN_BUFFER` / small `Iorec`) when it circulates the ring, dropping bytes at "start game." Fix = a **gated, orchestrator-only** smart layer (election coordination + ring flow-control), keeping the RP/byte-pipe dumb (D-02) and the dumb relay the default. The companion latency fix (`BUS_LOOP_MS` 100→1 ms, C-01) is what made these visible. _Empirically confirmed by `--inspect` traces: an all-slave `0x00` storm; a `COUNT-PLAYERS` interrupted by stray `0x00`s; and a desynced `80 84` (count value lost → START-GAME read as the count → "132 players")._ | accepted |

## Constraints

| ID | Constraint |
| --- | --- |
| C-01 | **Latency/jitter budget — MIDI Maze is lock-step.** Per §2.4.2, each machine must receive every other player's byte before computing the next frame, and *"la velocidad del juego (FPS) queda limitada por la velocidad del anillo"* — so network latency **directly sets the frame rate**, not just responsiveness. Worse, every write is followed by a synchronous MIDI-IN readback (`$188f0`, D-05) whose result aborts on negative, so a byte that arrives late can **freeze or desync** the game, not merely slow it. The budget is per-hop and must be at least as fast as the original physical MIDI ring-speed-per-hop (~hundreds of µs at 31250 baud + processing — which we beat by intercepting before the ACIA; see EPIC-07 STORY-01). First-class requirement: drives ring sizing (EPIC-02), send coalescing (EPIC-03), and the readback-servicing path (EPIC-01 STORY-04). |
