# Decisions

Lightweight log of cross-cutting decisions referenced by the epics/stories.
Status: **accepted** (decided) · **proposed** (leaning, needs validation) ·
**open** (undecided, needs a spike). Link decisions from stories as `D-NN`.

| ID | Decision | Status |
| --- | --- | --- |
| D-01 | **Target app is MIDI Maze.** The supported class is ST software that does MIDI I/O through the OS (`Midiws`, `Bconout`/`Bcostat`, `Bconin`/`Bconstat`, possibly `Iorec`). Apps that talk to the 6850 MIDI ACIA registers (`$FFFC04`/`$FFFC06`) directly and install their own MIDI interrupt handler are **out of scope**: those bus cycles never reach the cartridge, so this approach cannot intercept them. _Confirmed by the project's NotebookLM source, and refined on hardware: MIDI Maze accesses MIDI via BIOS (trap #13) Device 3 as a simple serial port._ | accepted |
| D-02 | **The MIDI stream is an opaque raw byte pipe.** MIDI Maze uses MIDI as a dumb byte transport, so we forward bytes byte-by-byte / buffered with **no MIDI parsing**: no message framing, no running-status handling, no active-sensing/clock filtering, no RTP-MIDI. Byte-exact, order-preserving passthrough. Any "be smart about MIDI" optimization would corrupt the game. _Confirmed by source: MIDI Maze sends raw 8-bit bytes (e.g. `0x00` designates the ring MASTER), not valid MIDI messages._ | accepted |
| D-03 | **Network transport = TCP with `TCP_NODELAY`.** The raw byte ring is order- and integrity-sensitive: a dropped/reordered byte desyncs the game, and UDP would mean reinventing sequencing/retransmit. Stack fit: the Pico-W firmware already runs lwIP TCP (httpc/download), and the Python orchestrator serves many concurrent player sockets cleanly with `asyncio` TCP. `TCP_NODELAY` defeats Nagle to protect latency (C-01); revisit only if EPIC-07 shows it insufficient. | accepted |
| D-04 | **Topology = star via a central orchestrator that emulates the ring.** Each Atari + Pico-W connects to a Python orchestrator (**in this repo** under `orchestrator/`, EPIC-04, which revises the earlier "separate project" plan). The orchestrator is a **ring-order router, not a broadcaster**: it wires OUT(player N) → IN(player N+1) preserving order so the protocol works naturally (master election via `0x00` returning, `0x80` COUNT-PLAYERS incrementing around the loop). **Ring membership must be fixed before a game starts:** adding or dropping a player mid-game breaks COUNT-PLAYERS and the lock-step (C-01). This implies EPIC-03's transport needs ordered, per-peer delivery, not broadcast. MVP = 2 players through the orchestrator; the full ring (up to 16 entities, humans + drones) is post-MVP. The firmware-side interface is specified in [`ORCHESTRATOR-CONTRACT.md`](ORCHESTRATOR-CONTRACT.md); the orchestrator lives in this repo under `orchestrator/` (EPIC-04, Python 3 stdlib-only). | proposed |
| D-05 | **MIDI path = BIOS Device 3 only (`Bconstat` / `Bconin` / `Bconout`).** MIDI Maze does ALL its MIDI ring I/O through BIOS device-3 (`trap #13`): it **sends** via `Bconout(3)` (fn 3, `jsr $341b2`) and **reads** input in the game loop via `Bconstat(3)` / `Bconin(3)` (fns 1/2, `and.l #$ff,d0`), specifically the readback that drives master election (`tst d0`==0 → MASTER) and the per-frame state exchange. **The earlier hypothesis** (thesis §2.4.2 + Annex B, read in the Steem debugger) that an **XBIOS (`trap #14`) MIDI-IN read** / the system `Iorec(2)` buffer also sat on the critical path was **disproven during hardware bring-up**: MIDI Maze touches neither; it is pure BIOS device-3. So our hook *is* the MIDI device: it answers `Bconin` / `Bconstat` directly from the RP's network IN queue (no system Iorec) and ships `Bconout` to the OUT path. `Midiws` (XBIOS 12) is not used. Source: <https://zaguan.unizar.es/record/146840/files/TAZ-TFG-2022-4616.pdf> §2.4.2 + Annex B (corrected on hardware). **Confirmed on hardware:** a tracer caught the first `Bconout(3)` byte = `0x00` (the master-election request), and a loopback let MIDI Maze read its own output back via `Bconin(3)` and become MASTER. | accepted |
| D-06 | **Validation harness is cartridge-resident.** The self-test runs as m68k code in the cartridge at boot (gated to a test build), not a standalone `.PRG`. | accepted |
| D-07 | **Physical-MIDI passthrough (THRU to the ST's own MIDI port) is deferred.** MIDI-to-IP replaces the physical link for MIDI Maze; local echo to the ACIA is not in the alpha scope. | accepted |
| D-08 | **Orchestrator handles raw bytes over sockets, never via a host MIDI driver.** A documented prior attempt to bridge the ST to a PC failed because host MIDI stacks (Windows `WINMM.dll`) filter/block non-standard raw single-byte traffic, with no API to pass a lone raw byte. Our design sidesteps this: the Pico-W intercepts at the Atari BIOS (device-3) layer and ships raw bytes over TCP to a Python (`asyncio`) socket server. The orchestrator treats the stream as opaque bytes; it must not route through any OS MIDI API. | accepted |
| D-09 | **MIDI Maze needs ≥2 players; a loopback can't start a game.** Confirmed on hardware: the MASTER reaches the config screen and then waits for a SLAVE to join the ring. A single node never starts a match. So the EPIC-01/02 loopback validates the *transport* (master election + COUNT-PLAYERS + config round-trip), not *gameplay*. Real gameplay needs a 2nd node: the orchestrator (EPIC-04) wires the ring, and the 2nd node is another ST+RP or, with no hardware, a **Hatari gateway player** (EPIC-05). The RP/byte-pipe stays protocol-dumb (D-02); any future "fake SLAVE"/drone is MIDI-Maze-aware logic that lives in the orchestrator (a later smart epic), never the RP. | accepted |
| D-10 | **Server + Hatari gateway live in this repo, Python 3 stdlib only.** The orchestrator (`orchestrator/`, EPIC-04) and the Hatari gateway (`hatari-gateway/`, EPIC-05) are written in Python 3 using only the standard library (`asyncio`, `socket`, `http.server`, `os.mkfifo`). No third-party packages, ever. The orchestrator MVP is a **dumb ring relay with low-level connection tracking**: no MIDI-Maze protocol parsing; game-session detection ("gameplays disputing/disputed") is a later smart epic. The Hatari gateway is a **software RP2040** (file-FIFO ↔ orchestrator) and the project's test peer (replaces the old desktop-test-peer story). | accepted |
| D-11 | **A real match needs a protocol-aware coordination layer in the orchestrator (EPIC-08); the dumb relay (D-10) is necessary but not sufficient.** Hardware testing (EPIC-05), once transport and latency were fixed, surfaced two blockers a byte-dumb relay can't solve: (a) **master election doesn't converge** when machines launch asynchronously: multiple nodes auto-elect, a SLAVE re-sends every `0x00` (Anexo A), and with no single master the election byte loops forever or re-triggers mid-`COUNT-PLAYERS`, so the MASTER reads a garbage/zero player count and never starts; (b) **the ~4 KB `SEND-DATA` maze (`0x83`) overflows the RP's IN path** (the small 1 KB IN queue / 256 B `MIDI_IN_BUFFER`, since enlarged to 16 KB) when it circulates the ring, dropping bytes at "start game." The fix is a **gated, orchestrator-only** smart layer (election coordination + ring flow-control), keeping the RP/byte-pipe dumb (D-02) and the dumb relay the default. The companion latency fix (`BUS_LOOP_MS` 100→1 ms, C-01) is what made these blockers visible. _Empirically confirmed by `--inspect` traces: an all-slave `0x00` storm; a `COUNT-PLAYERS` interrupted by stray `0x00`s; and a desynced `80 84` (count value lost → START-GAME read as the count → "132 players")._ | accepted |
| D-12 | **The per-byte m68k↔RP command handshake is the throughput ceiling: Iteration 1's transport must be redesigned (Iteration 2).** Iteration 1 proved the whole stack end-to-end, but hardware measurement (EPIC-07 STORY-01) found a fundamental limit the architecture can't tune away: every MIDI byte is a full `send_sync` command (write command+token, poll for the response), and that round-trip costs **~0.5 ms regardless of how fast the RP services it**. Measured **~970 bytes/s** each way (`MIDI cmd/s` counter), and making `chandler_loop` run every iteration (eager, vs the ~1 ms `BUS_LOOP_MS` gate) **did not move it**: the gate is the per-command bus handshake, not the RP loop cadence. A real MIDI ring is 31250 baud ≈ **3125 bytes/s**, so we are **~3× slower than the original 1987 hardware**. The ~4 KB `SEND-DATA` maze then takes ~4 s and gameplay can't keep lock-step (C-01). The 16 KB IN queue (EPIC-08 STORY-03) stops byte _loss_ on the burst but not the slowness. **Fix direction (Iteration 2):** stop doing a handshake per byte. Use shared-memory rings the m68k reads/writes directly in the cartridge window (owning its own read index), and/or batching many bytes per command; this needs a few bytes of m68k-side RAM for the index (cartridge code is ROM). _Empirically: `MIDI cmd/s: SEND≈RECV≈970` at peak, ramping then dropping as a transfer runs._ | accepted |
| D-13 | **WebSocket is an optional transport alongside TCP; TCP stays the default.** A node can carry its byte stream over WebSocket (RFC 6455) instead of the raw TCP socket (D-03). The node selects the transport (a persisted firmware/gateway setting, default `tcp`), and stores a port per carrier since the orchestrator's two listeners bind different ports (the toggle picks the carrier and its matching port together); the orchestrator enables an extra WebSocket listener via a CLI parameter (default off), running both listeners at once on different ports. A TCP node and a WebSocket node register into the same ring and relay to each other: the relay is transport-agnostic, so framing is transport-level only and the byte stream stays opaque (D-02), in-order, byte-exact. **Why:** raw TCP on a custom port (5005) is often blocked by firewalls and cannot pass an HTTP reverse proxy or many cloud frontends; WebSocket rides a standard HTTP port and an Upgrade handshake, so a node can reach an orchestrator hosted behind nginx, Cloudflare, or a PaaS load balancer. **Profile:** a GET Upgrade handshake on a configurable path (default `/`); opaque MIDI bytes in binary frames; client-to-server frames masked, server-to-client frames unmasked; ping answered with pong; close handled by reconnect. **Out of scope:** `wss` / TLS. The RP has no mbedTLS linked (`lwipopts.h` `LWIP_ALTCP_TLS=0`), so a secure socket is a separate, larger effort; operators who need TLS terminate it at a reverse proxy and speak `ws` to the orchestrator on the internal network. See [`ORCHESTRATOR-CONTRACT.md`](ORCHESTRATOR-CONTRACT.md) (Transport section) and EPIC-13. | accepted |
| D-14 | **A room key defines a private MIDI ring; one orchestrator hosts many.** A room key splits the single global ring into many private rings: players that present the same key share one ring, isolated from other rooms (room key `DIEGOROOM` gives everyone who typed it their own ring). The key rides the WebSocket handshake as `Authorization: Bearer <roomkey>` (D-13 carrier); a plain-TCP node has no handshake header, so it joins a single default room. The key is human-typed: case-insensitive (normalized to uppercase), `A-Z 0-9`, 1 to 16 characters (fits `SETTINGS_MAX_VALUE_LENGTH` and the firmware menu); an empty key means the default room. Rooms are **pre-provisioned** by an operator through a REST API on the HTTP status port: `GET /rooms` (open list), `POST /rooms` (create, admin), `DELETE /rooms/{key}` (delete, admin); writes require an `X-Admin-Key` header matching `--admin-key` and are refused when it is unset, reads are open. A WS join with an unprovisioned key is refused at the handshake (HTTP 403); the default room is always present. The orchestrator keeps one ring per room (replacing the single ring) and relays only within a room (D-04); dedup and reconnection recycling operate within a room, and telemetry is room-scoped. A room caps at 16 players (D-04, the MIDI Maze limit); `POST` may mint a short auto code or take a named key; an empty non-default room is reaped after a TTL; the provisioned room set persists to a JSON file across restarts. The web view scopes to one room (`status.json?room=KEY`) with a selector and a lobby, and badges each room's phase and master node from the read-only inspector (D-02 preserved). **Why:** one orchestrator can host many independent matches, and a typed key lets a group form a private ring with no per-player accounts. **Security:** the key gates a ring, it does not secure traffic; without TLS (`wss` deferred, D-13) the key and the MIDI bytes travel in clear text, so terminate TLS and guard the admin routes at a reverse proxy for an exposed deployment. See [`ORCHESTRATOR-CONTRACT.md`](ORCHESTRATOR-CONTRACT.md) (Rooms section) and EPIC-14. | accepted |

## Constraints

| ID | Constraint |
| --- | --- |
| C-01 | **Latency/jitter budget: MIDI Maze is lock-step.** Per §2.4.2, each machine must receive every other player's byte before computing the next frame, and *"la velocidad del juego (FPS) queda limitada por la velocidad del anillo"*, so network latency **directly sets the frame rate**, not just responsiveness. Worse, every write is followed by a synchronous MIDI-IN readback (`$188f0`, D-05) whose result aborts on negative, so a byte that arrives late can **freeze or desync** the game, not merely slow it. The budget is per-hop and must be at least as fast as the original physical MIDI ring-speed-per-hop (~hundreds of µs at 31250 baud + processing, which we beat by intercepting before the ACIA; see EPIC-07 STORY-01). First-class requirement: drives ring sizing (EPIC-02), send coalescing (EPIC-03), and the readback-servicing path (EPIC-01 STORY-04). |
