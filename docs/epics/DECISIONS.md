# Decisions

Lightweight log of cross-cutting decisions referenced by the epics/stories.
Status: **accepted** (decided) ┬╖ **proposed** (leaning, needs validation) ┬╖
**open** (undecided, needs a spike). Link decisions from stories as `D-NN`.

| ID | Decision | Status |
| --- | --- | --- |
| D-01 | **Target app is MIDI Maze.** The supported class is ST software that does MIDI I/O through the OS (`Midiws`, `Bconout`/`Bcostat`, `Bconin`/`Bconstat`, possibly `Iorec`). Apps that talk to the 6850 MIDI ACIA registers (`$FFFC04`/`$FFFC06`) directly and install their own MIDI interrupt handler are **out of scope** тАФ those bus cycles never reach the cartridge, so this approach cannot intercept them. _Confirmed by the project's NotebookLM source, and refined on hardware: MIDI Maze accesses MIDI via BIOS (trap #13) Device 3 as a simple serial port._ | accepted |
| D-02 | **The MIDI stream is an opaque raw byte pipe.** MIDI Maze uses MIDI as a dumb byte transport, so we forward bytes byte-by-byte / buffered with **no MIDI parsing**: no message framing, no running-status handling, no active-sensing/clock filtering, no RTP-MIDI. Byte-exact, order-preserving passthrough. Any "be smart about MIDI" optimization would corrupt the game. _Confirmed by source: MIDI Maze sends raw 8-bit bytes (e.g. `0x00` designates the ring MASTER), not valid MIDI messages._ | accepted |
| D-03 | **Network transport = TCP with `TCP_NODELAY`.** The raw byte ring is order- and integrity-sensitive тАФ a dropped/reordered byte desyncs the game; UDP would mean reinventing sequencing/retransmit. Stack fit: the Pico-W firmware already runs lwIP TCP (httpc/download), and the Python orchestrator serves many concurrent player sockets cleanly with `asyncio` TCP. `TCP_NODELAY` defeats Nagle to protect latency (C-01); revisit only if EPIC-07 shows it insufficient. | accepted |
| D-04 | **Topology = star via a central orchestrator that emulates the ring.** Each Atari + Pico-W connects to a Python orchestrator (**in this repo** under `orchestrator/`, EPIC-04 тАФ revises the earlier "separate project" plan). The orchestrator is a **ring-order router, not a broadcaster**: it wires OUT(player N) тЖТ IN(player N+1) preserving order so the protocol works naturally (master election via `0x00` returning, `0x80` COUNT-PLAYERS incrementing around the loop). **Ring membership must be fixed before a game starts** тАФ you can't add/drop a player mid-game without breaking COUNT-PLAYERS and the lock-step (C-01). This implies EPIC-03's transport needs ordered, per-peer delivery, not broadcast. MVP = 2 players through the orchestrator; the full ring (up to 16 entities, humans + drones) is post-MVP. The firmware-side interface is specified in [`ORCHESTRATOR-CONTRACT.md`](ORCHESTRATOR-CONTRACT.md); the orchestrator lives in this repo under `orchestrator/` (EPIC-04, Python 3 stdlib-only). | proposed |
| D-05 | **MIDI path = BIOS Device 3 only (`Bconstat` / `Bconin` / `Bconout`).** MIDI Maze does ALL its MIDI ring I/O through BIOS device-3 (`trap #13`): it **sends** via `Bconout(3)` (fn 3, `jsr $341b2`) and **reads** input in the game loop via `Bconstat(3)` / `Bconin(3)` (fns 1/2, `and.l #$ff,d0`) тАФ the readback that drives master election (`tst d0`==0 тЖТ MASTER) and the per-frame state exchange. **The earlier hypothesis** (thesis ┬з2.4.2 + Annex B, read in the Steem debugger) that an **XBIOS (`trap #14`) MIDI-IN read** / the system `Iorec(2)` buffer also sat on the critical path was **disproven during hardware bring-up** тАФ MIDI Maze touches neither; it is pure BIOS device-3. So our hook *is* the MIDI device: it answers `Bconin` / `Bconstat` directly from the RP's network IN queue (no system Iorec) and ships `Bconout` to the OUT path. `Midiws` (XBIOS 12) is not used. Source: <https://zaguan.unizar.es/record/146840/files/TAZ-TFG-2022-4616.pdf> ┬з2.4.2 + Annex B (corrected on hardware). **Confirmed on hardware:** a tracer caught the first `Bconout(3)` byte = `0x00` (the master-election request), and a loopback let MIDI Maze read its own output back via `Bconin(3)` and become MASTER. | accepted |
| D-06 | **Validation harness is cartridge-resident.** The self-test runs as m68k code in the cartridge at boot (gated to a test build), not a standalone `.PRG`. | accepted |
| D-07 | **Physical-MIDI passthrough (THRU to the ST's own MIDI port) is deferred.** MIDI-to-IP replaces the physical link for MIDI Maze; local echo to the ACIA is not in the alpha scope. | accepted |
| D-08 | **Orchestrator handles raw bytes over sockets тАФ never via a host MIDI driver.** A documented prior attempt to bridge the ST to a PC failed because host MIDI stacks (Windows `WINMM.dll`) filter/block non-standard raw single-byte traffic, with no API to pass a lone raw byte. Our design sidesteps this: the Pico-W intercepts at the Atari BIOS (device-3) layer and ships raw bytes over TCP to a Python (`asyncio`) socket server. The orchestrator treats the stream as opaque bytes; it must not route through any OS MIDI API. | accepted |
| D-09 | **MIDI Maze needs тЙе2 players; a loopback can't start a game.** Confirmed on hardware: the MASTER reaches the config screen and then waits for a SLAVE to join the ring тАФ a single node never starts a match. So the EPIC-01/02 loopback validates the *transport* (master election + COUNT-PLAYERS + config round-trip), not *gameplay*. Real gameplay needs a 2nd node: the orchestrator (EPIC-04) wires the ring, and the 2nd node is another ST+RP or тАФ with no hardware тАФ a **Hatari gateway player** (EPIC-05). The RP/byte-pipe stays protocol-dumb (D-02); any future "fake SLAVE"/drone is MIDI-Maze-aware logic that lives in the orchestrator (a later smart epic), never the RP. | accepted |
| D-10 | **Server + Hatari gateway live in this repo, Python 3 stdlib only.** The orchestrator (`orchestrator/`, EPIC-04) and the Hatari gateway (`hatari-gateway/`, EPIC-05) are written in Python 3 using only the standard library тАФ no third-party packages, ever (`asyncio`, `socket`, `http.server`, `os.mkfifo`). The orchestrator MVP is a **dumb ring relay with low-level connection tracking** тАФ no MIDI-Maze protocol parsing; game-session detection ("gameplays disputing/disputed") is a later smart epic. The Hatari gateway is a **software RP2040** (file-FIFO тЖФ orchestrator) and the project's test peer (replaces the old desktop-test-peer story). | accepted |
| D-11 | **A real match needs a protocol-aware coordination layer in the orchestrator (EPIC-08); the dumb relay (D-10) is necessary but not sufficient.** Hardware testing (EPIC-05), once transport and latency were fixed, surfaced two blockers a byte-dumb relay can't solve: (a) **master election doesn't converge** when machines launch asynchronously тАФ multiple nodes auto-elect, a SLAVE re-sends every `0x00` (Anexo A), and with no single master the election byte loops forever or re-triggers mid-`COUNT-PLAYERS`, so the MASTER reads a garbage/zero player count and never starts; (b) **the ~4 KB `SEND-DATA` maze (`0x83`) overflows the RP's IN path** (the small 1 KB IN queue / 256 B `MIDI_IN_BUFFER`, since enlarged to 16 KB) when it circulates the ring, dropping bytes at "start game." Fix = a **gated, orchestrator-only** smart layer (election coordination + ring flow-control), keeping the RP/byte-pipe dumb (D-02) and the dumb relay the default. The companion latency fix (`BUS_LOOP_MS` 100тЖТ1 ms, C-01) is what made these visible. _Empirically confirmed by `--inspect` traces: an all-slave `0x00` storm; a `COUNT-PLAYERS` interrupted by stray `0x00`s; and a desynced `80 84` (count value lost тЖТ START-GAME read as the count тЖТ "132 players")._ | accepted |
| D-12 | **The per-byte m68kтЖФRP command handshake is the throughput ceiling тАФ Iteration 1's transport must be redesigned (Iteration 2).** Iteration 1 proved the whole stack end-to-end, but hardware measurement (EPIC-07 STORY-01) found a _fundamental_ limit the architecture can't tune away: every MIDI byte is a full `send_sync` command (write command+token, poll for the response), and that round-trip costs **~0.5 ms regardless of how fast the RP services it**. Measured **~970 bytes/s** each way (`MIDI cmd/s` counter), and making `chandler_loop` run every iteration (eager, vs the ~1 ms `BUS_LOOP_MS` gate) **did not move it** тАФ proving the gate is the per-command bus handshake, not the RP loop cadence. A real MIDI ring is 31250 baud тЙИ **3125 bytes/s**, so we are **~3├Ч slower than the original 1987 hardware**; the ~4 KB `SEND-DATA` maze then takes ~4 s and gameplay can't keep lock-step (C-01). The 16 KB IN queue (EPIC-08 STORY-03) stops byte _loss_ on the burst but not the slowness. **Fix direction (Iteration 2):** stop doing a handshake per byte тАФ shared-memory rings the m68k reads/writes directly in the cartridge window (owning its own read index), and/or batching many bytes per command; this needs a few bytes of m68k-side RAM for the index (cartridge code is ROM). _Empirically: `MIDI cmd/s: SENDтЙИRECVтЙИ970` at peak, ramping then dropping as a transfer runs._ | accepted |

## Constraints

| ID | Constraint |
| --- | --- |
| C-01 | **Latency/jitter budget тАФ MIDI Maze is lock-step.** Per ┬з2.4.2, each machine must receive every other player's byte before computing the next frame, and *"la velocidad del juego (FPS) queda limitada por la velocidad del anillo"* тАФ so network latency **directly sets the frame rate**, not just responsiveness. Worse, every write is followed by a synchronous MIDI-IN readback (`$188f0`, D-05) whose result aborts on negative, so a byte that arrives late can **freeze or desync** the game, not merely slow it. The budget is per-hop and must be at least as fast as the original physical MIDI ring-speed-per-hop (~hundreds of ┬╡s at 31250 baud + processing тАФ which we beat by intercepting before the ACIA; see EPIC-07 STORY-01). First-class requirement: drives ring sizing (EPIC-02), send coalescing (EPIC-03), and the readback-servicing path (EPIC-01 STORY-04). |
